# 在庫発注最適化アシスタント

このプロジェクトは、単なる在庫管理ではなく、`いつ・何を・どれだけ発注すべきか` を分かりやすく示す在庫発注最適化サービスの初期プロトタイプです。

現時点では、CSVをアップロードして在庫状況を計算し、チャット形式で発注判断を確認できる `Streamlit` アプリとして動作します。

## 目的

目指しているのは「在庫を記録するツール」ではなく、「日々の発注判断を支援するサービス」です。

機械学習や複雑な数理最適化は一旦使わず、説明しやすく、現場でも扱いやすいシンプルなロジックで価値を出すことを重視しています。

このプロトタイプでは、以下のような問いに答えられることを目標にしています。

- どの商品が発注必要か
- 発注優先度が高い商品は何か
- なぜその発注量になるのか
- 安全在庫日数を変えるとどう変わるか

## 現在できること

- CSVアップロード
- 必須列のバリデーション
- 数値変換エラーの検知
- 実データ向けCSV正規化
- 安全在庫、発注点、推奨発注数、在庫日数の計算
- 全商品一覧の表示
- 発注が必要な商品の一覧表示
- 発注推奨リストのCSVダウンロード
- チャットUIによる在庫質問
- 安全在庫日数の簡易シミュレーション
- OpenAI API を使った GPT チャット連携

## 現在の計算ロジック

このアプリでは、シンプルで説明可能なルールベースの計算を使っています。

### 安全在庫

`safety_stock = avg_daily_sales × safety_days`

### 発注点

`reorder_point = avg_daily_sales × lead_time_days + safety_stock`

### 推奨発注量

`recommended_order = max(0, reorder_point - current_stock)`

表示時には切り上げ整数にしています。

### 在庫切れまでの日数

`days_left = current_stock / avg_daily_sales`

`avg_daily_sales` が 0 の場合は無限大として扱います。

### 発注要否

`need_order = recommended_order > 0`

## 入力CSV仕様

CSVは1商品1行で、以下の列が必要です。

- `product_id`
- `product_name`
- `current_stock`
- `avg_daily_sales`
- `lead_time_days`
- `safety_days`

### サンプル

```csv
product_id,product_name,current_stock,avg_daily_sales,lead_time_days,safety_days
1,ミネラルウォーター,20,3.5,5,3
2,お茶,8,2.0,7,2
3,コーヒー,50,1.2,10,5
4,カップ麺,5,4.0,3,2
```

### 実データCSVへの対応

実運用でよくある軽微な揺れを吸収するため、読み込み時にCSV正規化を行います。

- `utf-8-sig`、`utf-8`、`cp932` の読み込みを順に試します
- `,`、`;`、タブ区切りを吸収します
- `商品名`、`在庫数`、`リードタイム` などの列名を標準列へ寄せます
- `1,000`、`3日`、`¥500` のような値を数値変換しやすい形に整えます
- 空文字、`-`、`N/A`、`null` などを欠損として扱います

一方で、帳票形式のCSVや複数ヘッダ行のような複雑なファイルは対象外です。

```bash
pip install -r requirements.txt
```

### 3. アプリを起動

```bash

```

```bash

```

## GPT API 連携

このアプリは、通常はルールベースのチャットで動作します。加えて、OpenAI API キーを設定すると、サイドバーから GPT チャット連携を有効化できます。

### できること

- 在庫サマリーの自然文回答
- 商品情報の検索
- 予算付き発注案の取得
- 欠品リスク商品の抽出
- 安全在庫日数のシミュレーション

GPT は計算そのものを自由生成しているのではなく、アプリ側の在庫データと関数呼び出し結果をもとに回答します。

### 設定方法

方法1: サイドバーで直接入力

- アプリ起動後、サイドバーの `OpenAI APIキー` にキーを入力
- `GPTチャットを使う` をオンにする

方法2: 環境変数を使う

```bash
export OPENAI_API_KEY="sk-..."
streamlit run app.py
```

方法3: `secrets.toml` を使う

`.streamlit/secrets.toml` に以下を設定します。

```toml
OPENAI_API_KEY = "sk-..."
```

### キー未設定時の挙動

- GPT チャットは無効になります
- ルールベースのチャット機能はそのまま使えます
- CSVの計算、一覧表示、ダウンロード機能も通常通り動きます

### 使用上のメモ

- 現在のコードでは `openai` パッケージが必要です
- モデル設定は `app.py` 内の `OPENAI_MODEL` で管理しています
- API キーはリポジトリに直接書かず、環境変数か `secrets.toml` で管理する運用を推奨します

## このプロジェクトが目指すサービス像

このプロジェクトは、単なる在庫一覧画面ではなく、日々の発注意思決定を支援するサービスに育てることを想定しています。

つまり中心機能は「在庫を保存すること」ではなく、次のような判断を毎日出せることです。

- 今日発注すべき商品
- 今週欠品リスクが高い商品
- 予算内で優先すべき発注案
- なぜその商品を優先すべきか

## MVPとして次に入れたい機能

機械学習なしでも、以下を入れると一気にサービスらしくなります。

### 1. 商品ごとの発注条件

- 発注単位
- 最小発注数
- 原価
- 重要度
- 仕入先

### 2. 発注最適化ロジック

- 発注単位での丸め
- 最小発注数の適用
- 優先度スコアの計算
- 予算上限を加味した発注候補の選定

### 3. 判断支援UI

- 今日の推奨発注
- 欠品リスク一覧
- 過剰在庫一覧
- 予算内でのおすすめ発注案
- 商品ごとの発注理由表示

## シンプルな最適化の考え方

複雑な数理モデルを使わなくても、十分実務的な最適化は可能です。

基本の流れは次の通りです。

1. 各商品の推奨発注量を計算する
2. 発注単位や最小発注数で調整する
3. 欠品リスクや重要度に応じて優先度を付ける
4. 予算や制約の範囲で採用する
5. 最終的な発注案を一覧で出す

この方法はヒューリスティックですが、初期プロダクトとしては十分に強いです。
